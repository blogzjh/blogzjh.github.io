    1.场景

    • Metrics收集日志交给Graphite（Graphite本身也可以做为监控，但我们只是把他作为了一个数据存储，对grafana提供数据接口），Grafana做数据展示

    2.Metrics、Graphite、Grafana关系

    2.1Metrics

    • 日志收集系统，可以支持很多的监控系统
    • 一般在Spring项目中用其收集数据，可以发送到Graphite等监控系统中
    • 一般使用Merter和Timer分别记录成功率和响应时间指标

    2.2Graphite

    graphite是一套基于python的开源监控系统，主要包括三部分：

    • whisper数据库：树状文件存储结构（这在后面Grafana的分层级中能体现），Metrics的分层级（product . type . metriclog_key . metriclog_value）也会对应在whisper数据库中
    • carbon守护线程：负责从客户端接收数据，先在内存中做缓存，再存储至whisper（BufferPool的刷盘策略）
    • graphite-web展示页面：基于Django，同时读取whiper和carbon中为来得及持久化的数据

    2.3Grafana

    也是一套监控系统，使用效果比Graphite更好，因此可以通过Graphite做中间数据存储的桥梁（Metrics收集的日志不能直接到Grafana，Grafana能对接如es、graphite等数据源），最终展示在Grafana中

    3.Metrics工作原理

    对日志进行收集，一般分为响应时间记录和接口吞吐量：这里截取部分操作Metrics的代码，使用时调用封装好的log方法即可，其他配置细节忽略

    • 响应时间：Timer.update()* 吞吐量：Meter.mark()

    4.3.1三个注解

    记录响应时间，用于@Around

    @Retention(RetentionPolicy.RUNTIME)
@Target(ElementType.METHOD)
@Inherited
public @interface MetricLogTime {
    //metric-log key
    String key();
}

    通用的记录吞吐量，用于@AfterReturning和@AfterThrowing

    @Retention(RetentionPolicy.RUNTIME)
@Target(ElementType.METHOD)
@Inherited
public @interface MetricLogResult {
    //metric-log key
    String key();
    //如果是true，则把抛异常归为succeed而不是faild
    boolean exception() default false;
}

    Controller层的记录吞吐量，同上

    /**
 * 写这个注解的原因：AOP组件中@MetricLogResult在“@AfterReturnning”中，对于Controller层的返回值永远不为null
 * 本注解用途：专门用于对Controller层进行增强，可以判断ResponseResult.newSuccessResult(result)中的result是否为null
 * @author zjh
 * @create 2022-11-09 18:53
 */
@Retention(RetentionPolicy.RUNTIME)
@Target(ElementType.METHOD)
@Inherited
public @interface MetricLogControllerResult {
    //metric-log key
    String key();
    boolean exception() default false;
}

    4.3.2通用AspectJ

    @Component
@Aspect
@Slf4j
public class MetricLogAspect {
    
    //这是封装好了的Metric工具类，使用起来和普通日志记录一样
    @Autowired
    MetricLog metricLog;

    //这两个状态仅供Result吞吐量使用，用于监控成功率
    public static final String METRIC_SUCCESS = "succeed";
    public static final String METRIC_FAIL = "failed";

    @Around("@annotation(xxxxx.MetricLogTime)")
    public Object metricLogTime(ProceedingJoinPoint proceedingJoinPoint) throws Throwable {
        long start = System.currentTimeMillis();
        MethodSignature methodSignature = (MethodSignature) proceedingJoinPoint.getSignature();
        MetricLogTime metricLogTime = methodSignature.getMethod().getAnnotation(MetricLogTime.class);
        Object ret = proceedingJoinPoint.proceed();
        long end = System.currentTimeMillis();
        String key = metricLogTime.key();
        metricLog.log(key, end - start);
        return ret;
    }

    @AfterReturning(pointcut = "@annotation(xxxxx.MetricLogResult)", returning = "returnValue")
    public void metricLogType(JoinPoint joinPoint, Object returnValue) {
        MethodSignature methodSignature = (MethodSignature) joinPoint.getSignature();
        MetricLogResult metricLogResult = methodSignature.getMethod().getAnnotation(MetricLogResult.class);
        String key = metricLogResult.key();
        String val;
        if (returnValue == null) {
            val = METRIC_FAIL;
        } else {
            val = METRIC_SUCCESS;
        }
        metricLog.log(key, val);
    }

 //当且仅当@MetricResult的exception属性为true时，才执行这个@AfterThrowing
    @AfterThrowing(throwing = "ex", pointcut = "@annotation(xxxxx.MetricLogResult)")
    public void metricLogResultException(JoinPoint joinPoint, Throwable ex) {
        MethodSignature methodSignature = (MethodSignature) joinPoint.getSignature();
        MetricLogResult metricLogResult = methodSignature.getMethod().getAnnotation(MetricLogResult.class);
        String key = metricLogResult.key();
        boolean exp = metricLogResult.exception();
        String val;
         //当且仅当@MetricResult的exception属性为true时，才执行这个@AfterThrowing
        if (exp) {
            //这个异常是业务异常，不算失败
            if (ex instanceof ServiceException) {
                val = METRIC_SUCCESS;
            } else {
                val = METRIC_FAIL;
            }
            log.debug("@MetricLogResult {}={}", key, val);
            metricLog.log(key, val);
        }
    }
    }

    4.3.3针对Controller层的AOP

      /**
     * 补充这个AOP的原因：AOP组件中@MetricLogResult在“@AfterReturnning”中，对于Controller层的返回值永远不为null
     * 本@AfterReturning用途：专门用于对Controller层进行增强，可以判断ResponseResult.newSuccessResult(result)中的result是否为null
     *
     * @author zjh
     * @create 2022-11-09 19:00
     */
    @AfterReturning(pointcut = "@annotation(outfox.youdata.aop.annotation.MetricLogControllerResult)", returning = "returnValue")
    public void metricLogControllerType(JoinPoint joinPoint, Object returnValue) {
        MethodSignature methodSignature = (MethodSignature) joinPoint.getSignature();
        MetricLogControllerResult metricLogControllerResult = methodSignature.getMethod().getAnnotation(MetricLogControllerResult.class);
        String key = metricLogControllerResult.key();
        //这里不能直接用returnValue，需要取出ResponseResult.newSuccessResult(result)其中的result
        //如果开发者没有按照controller返回值封装json的规范，而直接return null，这里默认为成功
        String val = METRIC_SUCCESS;
        if (returnValue != null && returnValue instanceof ResponseResult) {
            ResponseResult returnValue2 = (ResponseResult) returnValue;
            Object data = ((ResponseResult<?>) returnValue2).getData();
            if (data == null) {
                val = METRIC_FAIL;
            }
        }
        metricLog.log(key, val);
    }

    @AfterThrowing(throwing = "ex", pointcut = "@annotation(outfox.youdata.aop.annotation.MetricLogControllerResult)")
    public void metricLogControllerResultException(JoinPoint joinPoint, Throwable ex) {
        MethodSignature methodSignature = (MethodSignature) joinPoint.getSignature();
        MetricLogControllerResult metricLogResult = methodSignature.getMethod().getAnnotation(MetricLogControllerResult.class);
        String key = metricLogResult.key();
        boolean exp = metricLogResult.exception();
        String val;
        if (exp) {
            //这个异常是业务异常，不算失败
            if (ex instanceof ServiceException) {
                val = METRIC_SUCCESS;
            } else {
                val = METRIC_FAIL;
            }
            log.debug("@MetricLogControllerResult {}={}", key, val);
            metricLog.log(key, val);
        }
    }

    5.Metrics与Grafana的日志层级关系

    像我们在Spring代码中用“@MetricsLogResult(key = “xxx”)、@MetricsLogTime(key = “xxx”)”进行埋点，启动Spring后Grafana不会出现任何相关信息

    5.1原因分析

    Metrics是收集日志到Graphite中，Grafana轮询Graphite的数据
    Metrics每次收集日志都会将其拼接为：项目组名.项目名.主机名.埋点名.埋点的可选值.metrics指标，再进行收集，因此只有当日志收集之后，Grafana那边才能选择类型

    5.2日志收集结构

    如上所说，日志的格式为：项目组名.项目名.主机名.埋点名.埋点的可选值.metrics指标

    • 一般来说，对于一个项目前两个（项目组名.项目名）比较固定，通常写在配置文件中直接写死
    • 如果Grafana接了集群的数据源，那么主机名就只会显示集群名字，集群的内部不可知
    • 埋点名就是AOP上那个key值，如果用Metrics收集日志时，把key值拼接为aaa.bbb那么就会再多出一个层级
    • （详细见另一篇Grafana）对于底层是Meter.mark()的埋点（有限离散）还可以选择value的值
    • （详细见另一篇Grafana）metrics指标详细

    6.Grafana使用

    • Grafana的数据来源于Graphite（由Metrics收集），其监控方式也是通过不停轮询（所以在Grafana中也叫新建query），如果Graphite中没有数据，Grafana中自然也不会显示，举个例子：
      1. 启动了Spring项目，但没有访问接口（此时接口被Metric记录），此时Grafana甚至都不知道是哪个“project”
      2. 请求接口，如果Metric记录了log(“myType . myMetric_key . myMetric_value1“) ，则只会在Grafana中出现value1的选项，不会出现value2（直到log记录到了value2）
    • 上述这个“Metric记录后才有选项”的特性十分重要，涉及到Grafana中的新建query是否有提示

    8.统计度量

    8.1Metrics统计度量类型

    Metrics提供了五种基本的度量类型：

    • Gauges：计量器
    • Counters：计数器
    • Histograms：直方图
    • Timers：计时器
    • Meters：TPS计数器

    一般来说最后两个满足了绝大部分监控的场景

    8.2适用场景

    8.2.1 Gauges

    Gauges中文名称可以翻译为计量器。它可以表示一个既可以增加, 又可以减少的度量指标值。它是最简单和最基本的Metrics类型，只有一个简单的返回值，**通常用来记录一些对象或者事物的瞬时值**。

    计量器的典型使用场景包括：

    温度
内存使用量

    8.2.2 Counters

    Counters中文名称可以翻译为计数器。它是一种累计型的度量指标，数值只能单调递增。

    计数器的典型使用场景包括：

    服务请求数
任务完成数
错误出现次数

    8.2.3 Histograms

    Histograms中文名称可以翻译为直方图。它可以用来度量流数据中value的分布情况，Histrogram可以计算最大/小值、平均值，方差，分位数（如中位数，或者95th分位数），如75%,90%,98%,99%的数据在哪个范围内。最大的特点是提供了多种多样的数据统计方式。

    直方图的典型使用场景包括：

    流量最大值
流量最小值
流量平均值
流量中位值

    8.2.4 Meters

    Meters中文名称可以翻译为TPS计数器。它是一种只能自增的计数器，通常用来度量一系列事件发生的比率(例如记录succeed和failed可以直到接口成功率)。他提供了平均速率，以及指数平滑平均速率，以及采样后的1分钟，5分钟，15分钟速率。

    TPS计数器的典型使用场景包括：

    平均每秒请求数
最近1分钟平均每秒请求数
最近5分钟平均每秒请求数
最近15分钟平均每秒请求数

    8.2.5 Timers

    Timers中文名称可以翻译为计时器。Histogram跟Meter的一个组合，另外还提供了对于某段代码执行的耗时的统计及计算功能。

    计时器的典型使用场景包括：

    请求时延
磁盘读时延

    8.4我司常用指标

    监控指标一般分为两大类：

    • 离散的有限状态：如：调用Metrics的meter.mark(k,v)，v的选择是有限且离散的（如succeed和failed）
    • 连续的无限状态：如：调用Metrics的指定key值对应的timer对象：timer.update(value,TimeUnit.MILLISECONDS)，value就是一个连续无限的“end-start时间”

    8.4.1 P系列指标解释

    p50, p90, p99（或者写作pct 50，pct90，pct 99）都是数据聚合统计一种方式，跟百分比相关（p的含义是percentile）。

    p50：数据集按升序排列，第50分位置大的数据（即升序排列后排在50%位置的数据）。
    p90：数据集按升序排列，第90分位置大的数据（即升序排列后排在90%位置的数据）。
    p99：数据集按升序排列，第99分位置大的数据（即升序排列后排在99%位置的数据）。
    P99.9：同上，精度到达99.9%，一般用P99和P99.9来展示绝大部分用户的接口响应时间，满足绝大部分用户的体验

    8.4.2 统计函数指标

    “stddev”: 方差

    “count”: 总次数

    “max”: 最长时间

    “mean”: 平均时间

    “min”: 最短时间

    8.4.3 比率相关（x分钟内平均）

    “m15_rate”: 15分钟 请求数/每秒的比率

    “m1_rate”: 1分钟内 请求数 / 每秒 即OPS

    “m5_rate”: 5分钟 请求数/每秒的比率

    “mean_rate”:平均每秒请求数

    “duration_units”: “seconds”,//该Timer的单位

    “rate_units”: “calls/second”//比率单位

    9.统计指标在Grafana中的体现

    9.1连续无限：timer.update()


    当我们记录了一条值，调用了一次这个方法，刚刚注册过的一堆指标都能选了

    9.2离散有限：meter.mark()

    但是如果调用meter.mark()，就只能看到一些离散的结果值，因为Metrics不会对这种值进行计算“平均值，百分值”等，想想也不合理嘛

    10.functions函数

    一般来说常用的就两个：sumSeries()和maxSeries()，用于观测整体status状态，host来自多个数据源时，通过聚合函数可以将其合并，达到观测整体的效果

    10.1maxSeries()监测整体最大延迟

    • 对于记录响应时间Timer.updata()的指标（P99、max、mean、m1_rate），如果Grafana同时监测了多个host的数据源，每次P99、max等“响应指标”都会有多个（host几个就几个），这时候需要使用maxSeries()来取最大值，来监测99%的用户最大延迟是多少

    10.2sumSeries()监测整体数量

    • 对于记录响应结果状态Meter.mark()的指标（m1_rate），这种指标只有数量一说，因此只需要通过sumSeries()达到聚合整体观测的效果
